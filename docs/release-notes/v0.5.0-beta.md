# pgrac v0.5.0-beta

**First public beta — the RAC core is in place.**

This is the first public release of pgrac: a shared-disk, multi-active cluster
built on PostgreSQL 16.13, following the Oracle Real Application Clusters model.
It is a **beta for evaluation and early pilots**, not a general-availability
release. See the [beta disclaimer](#beta-disclaimer) before deploying.

pgrac reaches this milestone with the seven core cluster capabilities working
end to end across nodes. What that means, how to run it, and where the edges
are — below.

---

## Added — RAC core capabilities

### Shared-disk multi-node cluster

Multiple PostgreSQL nodes open one shared database over shared storage. Nodes
are declared in `pgrac.conf` and brought up with the cluster bootstrap tooling.
A cluster-aware storage manager lets two instances open the same relation
concurrently.

### Cache Fusion block transfer

A block held dirty in one node's buffer cache is shipped directly to a
requesting node over the interconnect, coordinated through the global resource
directory, rather than forcing a disk round-trip.

### Cross-node MVCC

Snapshot visibility is resolved cluster-wide using a global SCN clock and
per-node undo, so a reader on one node sees a consistent view of writes
committed on another.

### Cluster crash recovery

When a node fails, its redo is replayed against shared storage so surviving
nodes converge on a consistent, committed state with no lost commits.

### Cooperative fencing / split-brain protection

The cluster runs fail-closed: a node writes only while a healthy majority of
voting disks confirms its membership. On quorum loss a node stops accepting
writes rather than risk divergence.

### Global Enqueue Service (GES) + cross-node deadlock detection

Table, sequence, advisory and other lock classes are coordinated across nodes
through a distributed enqueue service, with a cluster-wide deadlock detector
that finds and breaks cycles spanning multiple nodes.

### Online cluster reconfiguration

Nodes can join, leave cleanly, or be removed while the cluster stays up —
membership changes converge without a full-cluster restart, one reconfiguration
at a time.

### Observability

Cluster state is visible through SQL: cluster system views and 116
cluster-specific wait events surface in the standard `pg_stat_activity`
columns.

## Known limitations

These are user-facing edges of the beta. The full register, with the current
behavior for each, is in [docs/user-guide/beta.md](../user-guide/beta.md).

- **Node count** — up to 16 nodes.
- **Fencing** — membership fencing is cooperative and fail-closed at the
  database layer. Hardware-enforced isolation (STONITH / SCSI-3 persistent
  reservations) is not enforced by this beta; pair the cluster with an external
  node-level watchdog for isolation guarantees.
- **Not part of the validated beta surface** — RDMA interconnect transport,
  Active Data Guard standby, and backup / point-in-time recovery are present in
  the tree as experimental work and are **not covered by this beta's
  validation**; do not rely on them for evaluation.
- **Storage** — the validated configuration uses the shared-filesystem storage
  backend. A block-device backend exists but its production hardening is not
  part of this beta.
- **Cross-node read cache** — the cross-node consistent-read cache fast path
  ships disabled by default; enable it only per the read-heavy tuning guidance.
- **Cross-node forward Cache Fusion** — some forward-transfer paths are
  partially limited.
- **Scale evaluation** — a single machine can run a small cluster for
  functional evaluation; use separate hosts to evaluate multi-node behavior
  under sustained load.

## Beta disclaimer

pgrac v0.5.0-beta delivers the RAC core for **evaluation and early pilots**. It
is **not** a general-availability release.

- **No service-level, uptime, or data-integrity guarantee.** Do not run
  irreplaceable production data on this beta without independent backups.
- **Not GA.** General availability arrives with a later `v1.0.0` release after
  production hardening.
- **Support model** — issues are tracked publicly and addressed with patches;
  there is no commercial SLA.
- **No upgrade path** — this is the first release; in-place upgrade from beta is
  not offered.

The `--disable-cluster` build remains binary-identical to upstream PostgreSQL
16.13 and passes the full 219-test regression suite, so the cluster code is a
strictly additive, opt-in path.
